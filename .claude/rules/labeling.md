# Labeling System

> A system that defines **abstract expressions** (location, element, color, target, etc.) spoken by users as **clear labels** to reduce communication errors between AI and users.

## Label Document Location

```
Docs/Labels/
├── LABEL_UI_LAYOUT.md      # Screen layout labels (areas, panels, sections)
├── LABEL_UI_COMPONENT.md   # UI component labels (buttons, inputs, cards)
├── LABEL_ARCHITECTURE.md   # Project structure labels (folders, files, modules)
├── LABEL_STATUS_MESSAGE.md # Status message labels (success, error, warning)
└── LABEL_AI_COMMUNICATION.md # Trigger-label conversion table
```

## AI Behavior Rules

### 1. Use Labels in Responses
- Convert user's abstract expressions to labels in responses
- Example: "right button" → `[SUBMIT_BTN]` in `[RIGHT_PANEL]`

### 2. Update Label Documents (Required)
**Update `Docs/Labels/` documents in these situations:**

| Situation | Action | Target Document |
|-----------|--------|-----------------|
| New UI element created | Add label | `LABEL_UI_LAYOUT.md` or `LABEL_UI_COMPONENT.md` |
| New folder/file structure | Add label | `LABEL_ARCHITECTURE.md` |
| New status message added | Add label | `LABEL_STATUS_MESSAGE.md` |
| User expression interpreted successfully | Add trigger-label mapping | `LABEL_AI_COMMUNICATION.md` |
| **2+ communication failures** | Must add/modify label | Related document |

### 3. Label Writing Rules
```
Format: [CATEGORY_NAME] or [PARENT]_[CHILD]
Example: [MAIN_WINDOW], [HEADER_LOGO], [SIDEBAR_MENU]
```
- Use uppercase + underscore
- Express hierarchy as parent_child format
- Meaning must be clearly conveyed
